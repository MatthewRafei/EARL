1. Introduction

EARL (Evaluate And Run Language) is being designed to be functionally similar to Python, except with types and brackets.

2. Compiling

EARL uses autoconf to generate Makefiles. The following are required to build the project:

  1. autoconf
  2. automake
  3. libtool

Users compiling for the first time should issue either 

cd EARL/src
./setup.sh

or 

cd EARL/src/
autoreconf --install
./configure

This will create the Makefiles. Use make <opt> where <opt> is one of,

  • make → builds the project
  • make test → build the project and runs tests
  • make clean → cleans the project

Now that the initial setup has been complete, you no longer have to use autoconf.

3. Contributing

3.1. Style Guide

3.1.1. Functions

  • Functions should be formatted as such:
<return type> <name>(<params>) {
  body...
}
  • All indents should be 4 spaces and do not use tabs.
  • Macros should be in CAPS. The only exception is for when you want to treat a macro as a function.
  • Always put a comment after #endif with what it is ending.

3.1.2. Structs

struct <NameOfStruct> {
  ...
}
  • All struct names must use pascal case <NameOfStruct>.

3.1.3. Misc

  • All if, for, while etc. should have brackets (the left bracket should be on the same line as the statement). While not required, if a conditional/loop only has one statement, you can ommit the brackets. Just make sure it is on a different line.
  • +, -, *, /, etc. does not need whitespace, but == should.
  • Lines have a soft limit of 85 characters.
  • typedef'd types should have _t appended to it. Example:
typedef int my_type_t;
  • All variable, function, and macro names should be snake_case.
  • All file names have a hyphen ('-') to separate words.
  • Remove unnecessary trailing whitespace.
  • Disable auto-formatters (unless it conforms with the C default style of BSD).

4.1. Test Suite Source

The test suite is composed of two files, test-suite.cpp and test-suite.hpp

The file test-suite.hpp contains a bunch of custom assert macros that return the correct values on failure. Use these when comparing values, checking for null, etc.

The file test-suite.cpp is the file where you actually issue running the tests. This is done with the #define RUN_TEST(func) macro where func is the test that you want to run.

All tests return a type of test_errno_t, and are all expected to return that. Additionally, the tests should not take any parameters. At the end of every test, the function should return TEST_OK.

The project is compiled with -I$(top_srcdir)/include so there's no need to have #include "../../include/" when needing headers.

If a test were to fail after running make test, it will output "FAILED (\(n\))" where \(n\) is some test assert failure number. To figure out what the number means, go into test/test-suite.hpp and match it to the appropriate TEST_ASSERT_*_FAILURE macro.

4.2. Adding more Tests to Ones that Exist

Simply go into the test directory that you want to add tests for and create a new function. Then add it's prototype to its corrosponding header file.

Lets say I want to add a new test to vec. I would go into test/vec/test-vec.hpp and add a new prototype 

#ifndef VEC_TEST_H
#define VEC_TEST_H

#include "test-suite.h"

test_errno_t test_vector_instantiation(void);
test_errno_t test_vector_insert_elems(void);
.
.
.
test_errno_t test_vector_my_new_test(void); // here

I would then go into test/vec/test-vec.cpp and add it's implementation 

test_errno_t test_vector_my_new_test(void) {
  struct vec v = vec_create2(int);
  ...
  TEST_ASSERT_EQ(vec_deref_at(v, 1, int), 99); // example of a test assert.
  ...
  vector_free(&v); // don't forget to free the memory!
  return TEST_OK;
}

Finally, go into test/test-suite.cpp and issue the macro to actually run the newly created test. 

int main(void) {
  int pass = 0;
  int fail = 0;

  time_t start, end;
  double elapsed;

  start = time(NULL);
  RUN_TEST(test_vector_instantiation());
  RUN_TEST(test_vector_insert_elems());
  RUN_TEST(test_vector_large_elements());
  ...
  ...
  ...
  RUN_TEST(test_vector_my_new_test()); // here
  ...
  ...
  ...
  elapsed = difftime(end, start);
  printf("[test] total time: %.5f\n", elapsed);
  printf("TEST SUMMARY\n");
  printf("  PASS: %-5d FAIL: %d\n", pass, fail);
}

That's it! Now run make test to see the output.

4.3. Creating New Tests not Associated with Ones that Exist

This is more involved and will likely change in the future. But for now, here is how to do it:

NOTE: The following steps are being performed in EARL/src.

NOTE: The following code snippets are "copy-and-paste-able."

NOTE: Anywhere that says testname, it is refering to the type of test group that you are creating.

  1. In test/, create a new directory for the type of tests you want to create. Let testname be the directory name.
  2. Go into test/Makefile.am *note the .am.
  3. In test_suite_SOURCES, add testname/test-testname.cpp

  4. Create the testname directory and add the .cpp and .hpp files. 

    mkdir test/testname
touch test/testname/test-testname{.cpp,.hpp}

  5. Add include guards to the header 

    cat > test/testname/test-testname.hpp << "EOF"
#ifndef TESTNAME_H
#define TESTNAME_H

#include "test-suite.hpp"

#endif // TESTNAME_H
EOF

  6. Add the appropriate headers to test/testname/test-testname.cpp 

    echo "#include \"test-suite.hpp\"" > test/testname/test-testname.cpp
  7. In test/test-suite.cpp add the appropriate include header.
  8. Start adding your tests (see Adding more Tests to Ones that Exist)

5. Interfaces

5.1. Arena

File: arena.h

Description: Provides an API for an arena allocator. This is helpful because we have a "pool" of memory that we can use and then free it all at once. If unfamiliar with arena allocators, https://en.wikipedia.org/wiki/Region-based_memory_management

5.1.1. Functions

struct arena;

The arena struct. 

struct arena *arena_create(size_t cap);

Create a new arena allocator with capacity as cap. 

uint8_t *arena_alloc(struct arena *arena, size_t bytes);

Allocate bytes number of bytes in an arena. 

void arena_free(struct arena *arena);

Free the memory inside of arena AS WELL AS THE ARENA ITSELF.

5.3. Notify DEPRECATED

File: notify.h

Description: Provides ways to display an error message and quite the program.

5.3.1. Enums

enum NOTIFY_ERR_TYPE

Provides different types of errors.

5.3.2. Macros

#define NOTIFY_ERR(etype, msg)

Macro that prints the error type etype as well as prints msg. It will exit with a status of 1. 

#define NOTIFY_ERRARGS(etype, msg, ...)

It is the exact same as NOTIFY_ERR except it takes a variadic number of arguments. This is useful if your message has format specifiers in it such as "%d", "%zu" etc.

5.4. Utils DEPRECATED

File: utils.h

Description: A bunch of helpful utility tools.

5.4.1. Macros

Debug assertions that are taken out when compiling in "release" mode. In "debug" mode, they will function.

A macro that void's x. This is useful to silence compiler warnings.

Used to silence compiler warning on a function with some return value that is not yet returning anything. It prints msg and returns retttype

5.4.2. Functions

A malloc wrapper that checks if malloc failed. Allocates bytes number of bytes.

A wrapper for strcmp that checks if the strings are equal. Returns 1 on success, and 0 on failure.

Fill an array from 0..len with values 0..len.

5.5. Vector DEPRECATED

File: vector.h

Description: A vector interface. It is a generic array that grows by \((\) vector.cap \(\times 2)\) when the length matches the capacity.

5.5.1. Structs

struct vector

The vector struct.

5.5.2. Macros

#define vector_create2(type)

Macro to make it clear what kind of element the vector is going to hold. It creates a vector that will hold the type

Example: struct vector v = vector_create2(int); 

#define vector_deref_at(v, idx, cast)

Gives you the element in the vector v at idx and casts it as cast. 

#define vector_debug_dump(v, formatstr, type)

Debug prints the vector v with the format string formatstr and casted as type. 

#define vector(type)

Only used when instantiating a new vector or dispalying its typey. This allows to explicitly state what it holds, however nothing happens if the type does not match what it is actually holding.

Example:

struct vector(int) v = vector_create(int);

struct F { struct vector(int) v, ... };

5.5.3. Functions

struct vector vector_create(size_t stride);

Create a vector where the size of each element to be stored is stride. 

void vector_append(struct vector *v, void *data);

Appends data to the END of the vector v. 

void *vector_at(struct vector *v, size_t i);

Retrieve an element at i. It is up to the caller to cast as necessary. 

void vector_free(struct vector *v);

Free the underlying data in v. Resets all member variables. 

void vector_rm_at(struct vector *v, size_t idx);

Remove an element at a specific index idx. 

uint8_t *vector_asbytes(struct vector *v);

Gives the underlying data as uint8_t.
